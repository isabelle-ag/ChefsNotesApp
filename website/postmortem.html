Postmortem

1. What was the overall architecture of your system (particularly if it is different from the demo system)?

Our architecture essentially matched the demo system. This included business, presentation, and persistence layers. Along with extra packages for objects and application services.

2. What went right in the development process?

Group cooperation and conflict presentation were of utmost importance. Any differences were settled immediately, and work was accomplished in a coordinated and communicative manner.

3. What went wrong in the development process?

Repeated procrastination, we had a tendency to push back the start of the current workload which resulted in periods of rushing near the end of the iteration.

4. What would you do differently, if you had the chance to start over?

We would begin building the core of the project, UI components, and dynamic shared objects during iteration 0. The course is quite short, and iteration 1 gave us little time to begin creating the project core in addition to the first set of features.

5. How large is the project (number of methods, classes, etc)? How much of this is devoted to each major system component? And any other quantifiables (e.g. recorded hours on tasks)?

- 2144 lines, about half of which are in presentation, the majority of the rest are in objects and persistence.
- 357 methods, primarily in presentation and persistence.
- 50 classes, about half of which are in presentation.

6. What took the most time? The least? Any surprises?

The most time was spent on developing UI components. This was not overly surprising, we had expected to encounter this as none of us had experience working in Android Studio before beginning this project. UI components are also hefty and require constant tweaking to ensure perfect alignment and spacing.

7. Are there any particular design smells, or brilliant design decisions?

A design smell that we would've liked to fix but ended up taking lower priority to new features is the size of the EditRecipe class. It has become quite monolithic in size and we believe it could be refactored into a few smaller classes for readability and maintainability. This is scheduled for a future change.

8. Are there any outstanding bugs?

Recent recipe gets set in the recipe browser erronously at times.

9. Did any features work better than expected?

The recipe browser turned out better than we had initally hoped for, it looks and feels very clean and easy to use.

10. Are you using any technologies other than what was required (e.g. JMock, GUI builders, etc.)?

Not for actual development, but we did use a Miro board for brainstorming and planning which turned out very well.
Link to the Miro

11. Are you using any specific techniques covered in the course (TDD, pair programming, scrums, etc)?

Other than the iteration based 'agile' method, no not really.

12. How did the project change from your inital (iteration 0) vision or stories, or did it work out as predicted?

We ended up having to drop some features, most notably the meals/meal planning stuff, which we're kinda bummed about. The loss of features was unfortunate, but given the opportunity for a forth iteration, we would have been able to implement them.

13. What did you learn about team or large project development? What will you start doing, keep doing, or stop doing next time?

Procrastination leads to more issues than it's worth. We'd love to say that we'll get started on things earlier in the future but honestly, who knows.

14. Can you draw any conclusions from what you've done?

- There is a lot more work to building a working project than we had initally expected.
- Teamwork, while not easy and often frustrating, is very rewarding.
- Communication and coordination within the team is the most important part of software design, even above principles, patterns, conventions, and designing.
